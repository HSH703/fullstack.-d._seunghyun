1. SQL 기본쿼리구문 (예시: 본인 포트폴리오에서 사용한 것들 기준)
- INSERT 작성 (데이터 추가)
    > dml: 데이터 조작어 - insert(데이터추가), update, select, delete
    > 정의 : 테이블에 새로운 데이터 추가 sql
    > 예시: insert into 테이블명(필드명1, 필드명2,,,,) values(필드값1, 필드값2,,,,);
   
    > pk가 중복되면 어떻게 되나요? 
     - pk: 각 데이터를 식별하는 키, 중복허용X -> sequence.nextval 이용해서 중복키 허용 안함.
     - insert시 오류 발생(unique오류)

- UPDATE 작성 (데이터 수정)
    > dml: 데이터 조작어 - insert, update(데이터수정), select, delete
    > 정의: 테이블에 데이터를 수정 sql
    > 예시: update set 필드명 where 필드명=필드값1; 

    > where 조건을 잘못주면? 전체데이터가 변경&데이터 삭제될 위험↑.

- JOIN 간단하게 작성 
    > dml: 데이터 조작어 - insert, update, select(데이터조회), delete
    > 정의: 두개 이상의 테이블을 연결 -> 하나의 테이블처럼 사용하는 조회 sql 
    > 예시: ooo을 만들었는데, 여기에서 oo 외래키를 기준으로 조회했습니다.
        select 
        from 테이블 u join 테이블 o on u.id = o.id;
    > join의 종류: inner join(일치하는 값-교집합), left, right, full
                  outer join(매칭되지 않을 경우 null 값 반환-합집합)
    > right join(right 모든데이터 가져오기) 
      full outer join(양쪽 테이블의 모든 데이터 가져오기 - 매칭이 안되면 null 반환)
    > join(다중테이블을 연결해서 복잡할때 유리)
      VS 
      subquery(특정조건에 맞는 데이터 가져올때 더 직관적)
```
INNER JOIN

SELECT A.필드명1, B.필드명2
FROM TableA AS A
INNER JOIN TableB AS B
ON A.id = B.a_id;
```
```
LEFT JOIN

SELECT A.column1, B.column2
FROM TableA AS A
LEFT JOIN TableB AS B
ON A.id = B.a_id;
```
```
RIGHT JOIN

SELECT A.column1, B.column2
FROM TableA AS A
RIGHT JOIN TableB AS B
ON A.id = B.a_id;
```
```
FULL OUTER JOIN

SELECT A.column1, B.column2
FROM TableA AS A
FULL OUTER JOIN TableB AS B
ON A.id = B.a_id;
```

- 추상화(공통개념뽑기), 상속(기존클래스 확장), 인터페이스(규약정의)
    > 추상화: [공통개념추출]/복잡한 시스템에서 핵심적인 특징만 뽑아내서 표현
    > 상속: [기존클래스 확장]/기존 클래스(부모)의 속성과 메서드를 새로운 클래스(자식)가 물려받아 사용하는 것.
    > 인터페이스: [규약정의]/기존 클래스(부모)의 속성과 메서드를 새로운 클래스(자식)가 물려받아 사용하는 것.
    > why? 객체지향의 핵심, 코드 재사용 및 유연성
    > 상속 과사용시 : 결합도가 높아서 유지보수 어려움.

- 추상클래스 VS 인터페이스 
    > 추상클래스(공통된 속성과 메서드 정의) VS 인터페이스(구현해야할 메서드의 규약제공) 
    > 상속을 통해서 확장가능                 다중 구현(상속/implements) 가능
    > 인터페이스에서 default 메서드 추가된 이유 
        - 기존사용하는 클래스들이 깨지지 않도록 하면서도 새로운 기능확장
    > 다중상속 해결? 
    다중 상속을 직접 허용X → 인터페이스 + default 메서드 + 조합(Composition)을 통해 우회적으로 해결.

- MVC패턴 (mvc1  vs  mvc2) 차이점 - controller의 역할 jsp, servlet이 하는지 구분.
    > MVC는 Model, View, Controller로 역할을 분리하는 패턴
    > mvc1: jsp가 controller와 view를 동시 수행, 
      mvc2: servlet이 controller역할
    > mvc2가 유지보수성 측면에서 나은 이유? 역할이 분리되어 코드 재사용성과 확장성↑.
    > spring mvc 구조는 어떻게 작동? (Dispatcher servlet)
        > Spring MVC는 Model-View-Controller 패턴을 기반으로 한 웹 애플리케이션 프레임워크입니다. 핵심은 모든 요청을 중앙에서 처리하는 DispatcherServlet(프론트 컨트롤러)에 있습니다. 


- 의존성 (DI) - 프레임워크가 전체흐름 제어/코드 호출하는 주체가 다름 (개발자 VS 프레임워크)
    > 프레임워크 : 개발 틀(재사용가능한 코드와 구조를 제공)
    > 라이브러리: (개발자가 필요할때 사용하는 도구모음/개발자가 코드호출)
      프레임워크: (프레임워크가 전체흐름 제어, 내 코드 호출)
    > ioc: 제어의 역전 - 객체의 생성부터 소멸까지 프레임워크가 제어권을 가짐.
        (mvc는  객체의 생성부터 소멸까지 개발자가 제어권을 가짐)
    > di: xml 객체간의 의존성을 외부에서 설정, 개발자가 단순히 의존성을 선언하면 컨테이너가 알아서 객체를 생성(팩토리패턴)하고 주입(di)해줌.
